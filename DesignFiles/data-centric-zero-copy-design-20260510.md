## Problem Statement

A ROS2 + DDS pipeline for Physical AI suffers two simultaneous problems:

1. **Data plane**: serialization/deserialization and memory copies inflate P99 latency, breaking the time-synchronization accuracy needed for high-bandwidth multimodal data (multiple cameras, LiDAR, IMU). This directly degrades VLA / robot-learning dataset quality.
2. **Control plane**: there is no way to deliver accelerator (NPU/GPU) inference results into an RT control loop with a **deterministic staleness bound**. The resulting non-determinism makes safety analysis impossible.

Existing ROS2 + Iceoryx (rmw_iceoryx_cpp) or Iceoryx2 integrations only solve zero-copy at the **message middleware level**. Even when a message reaches RAM zero-copy, another copy is required to move the payload onto the accelerator.

## What Makes This Cool

- **An integration layer where zero-copy never breaks from sensor → message → accelerator.** The message queue (Iceoryx2 SHM) carries only the **metadata** of a memory descriptor; the dma-buf **FD itself is delivered exactly once via an SCM_RIGHTS / pidfd_getfd sidecar** during the subscribe handshake. From then on the payload stays as the dma-buf exported by V4L2 capture and imported by the accelerator driver — never copied through host memory.
- **bounded staleness as a first-class property.** The staleness bound observed by the RT control loop is computed explicitly as the sum of **OS-level terms** (sensor capture, dma-buf fence wait, inference worst-case, metadata publish, seqlock retry, dma-buf view access) — usable directly in safety analysis.
- A single system that delivers both **data-plane zero-copy and control-plane determinism** in one closed-loop demo — attacking ROS2's two weaknesses at once.

## Constraints

- The first build targets a **single UMA or UMA-like platform** (locked in during the week 1-2 spike).
- We will **not** make the accelerator driver RT-aware. Inference runs in a non-RT worker.
- The shape of the project is a **minimal library / SDK**. A graph runtime / ROS2 RMW backend is a later phase.
- Audience: **VLA data-collection / training researchers + robot control engineers**, both.

## Premises (Agreed)

1. The differentiator is not message zero-copy but **"the end-to-end sensor→accelerator→control zero-copy integration layer."** Iceoryx2 is reused as-is for the metadata layer. dma-buf FDs are delivered via a separate **SCM_RIGHTS / `pidfd_getfd(2)` sidecar** during the subscribe handshake (with sidecar calls amortized through buffer-pool reuse).
2. The first build is a single platform only. **Apple Silicon is excluded** from the first build because macOS lacks V4L2 and Asahi Linux lacks an ANE driver. Candidates are **AMD AI Series (XDNA)** and **NVIDIA Jetson Orin (NVMM/EGLStream)**, locked in during the week 1-2 spike.
3. The first build is a closed-loop integrated demo (V4L2 → DMA-BUF → NPU/GPU → 1kHz RT loop). Data plane and control plane are not demoed separately.
4. The RT/accelerator integration follows the pattern **"inference is a non-RT worker; only the result is passed zero-copy into the RT loop."** Inference is variable and not deterministic, so it never runs in an RT context.
5. The audience covers VLA researchers + robot control engineers — a single demo shows two values (data-sync accuracy + RT determinism).

## Approaches Considered

### Approach A: Minimal Library / SDK (selected)
- Wraps Iceoryx2 with a `TensorDescriptor` payload + sidecar handshake + per-platform accelerator-import backend + RT consumer lock-free helpers.
- Effort: M (12-14 weeks; +2-4 weeks added by sidecar handshake and PoC spike after review). Risk: Med.

### Approach B: Graph Execution Runtime
- Pipelines defined as nodes-and-edges; the runtime auto-maps to RT/non-RT regions.
- Goes head-to-head with NVIDIA Holoscan SDK. Effort: XL (1y+). Risk: High.

### Approach C: ROS2 RMW Backend
- ROS2 nodes unchanged; only the topic transport is replaced with accelerator zero-copy.
- Adoption is the easiest, but the ROS2 message model does not naturally express an accelerator descriptor. Effort: M. Risk: Med.

## Recommended Approach: A

**Why:**
- Fastest path to validating the differentiator (end-to-end zero-copy + bounded staleness).
- The core abstraction (`TensorDescriptor` + FD sidecar) carries forward into B's graph node data model and C's RMW payload representation — **no evolutionary path is closed off**.
- No ROS2 dependency, so the audience is broadest. ROS2 users can still adopt via a thin wrapper.

### Core Components

1. **`TensorDescriptor` (Iceoryx2 payload = fixed-size metadata only)**
   - `bo_handle_t`: producer-allocated id used by the receiver to identify the FD that arrived through the sidecar
   - `seqno`: buffer version (seqlock + ABA prevention + staleness measurement)
   - `shape[N]`, `dtype`, `offset`, `size`: tensor shape
   - `capture_ts_ns`, `producer_publish_ts_ns`: timestamps (V4L2 hardware where available, otherwise monotonic)
   - `sync_fence_kind`: `none` / `sync_file_fd_via_sidecar` / `dma_resv_implicit`
   - `pool_generation`: detects buffer pool reallocation (triggers FD re-delivery)

2. **FD sidecar (Unix domain socket + `SCM_RIGHTS`)**
   - At the subscribe handshake the producer delivers every dma-buf FD in the buffer pool at once.
   - On a (rare) pool reallocation the `pool_generation` increments and a fresh batch of FDs is delivered.
   - The receiver supports a `pidfd_getfd(2)` fallback path (Linux 5.6+).
   - **Liveness**: at metadata publish time the receiver must already hold an FD cache for the relevant `pool_generation`. On miss the message is treated as stale and the receiver re-requests through the sidecar.

3. **Publisher / Subscriber wrappers**
   - `iceoryx_publisher<TensorDescriptor>` / `iceoryx_subscriber<TensorDescriptor>` are reused as-is.
   - The wrapper handles sidecar connection, handshake, FD-cache management, and lifecycle integration.

4. **Platform Backends (one per first build, locked in by the week 1-2 spike)**
   - **Candidate 1 — AMD AI Series (XDNA)**: V4L2 dma-buf export → AMDXDNA import. The key PoC question is whether external dma-buf import has a production path in the mainline driver.
   - **Candidate 2 — NVIDIA Jetson Orin**: V4L2 → NVMM dma-buf → cuMemImportFromFd / EGLStream. Vendor-specific, but the production path is well-traveled.
   - Apple Silicon is out of scope.

5. **RT consumer helpers**
   - **Pinning is mandatory**: `mlockall(MCL_CURRENT|MCL_FUTURE|MCL_ONFAULT)` + `MAP_POPULATE` + force a page fault at attach time.
   - Seqlock-based lock-free read returns a zero-copy view of the most recent descriptor.
   - **Stale-fallback policy**: when seqlock retry exceeds the cap (e.g. 8) the helper falls back to the last-known value or a zero command — the policy is injectable.
   - **Priority-inversion avoidance**: dma-buf attach/detach is performed only by the producer (the RT consumer receives an already-attached handle). The wrapper guarantees the RT consumer never touches `dma_resv` directly.

### Measurement Targets (Redefined)

**Worst-case staleness (each term measured separately and summed):**

```
staleness ≤ 
    sensor_capture_to_dma_buf_export_max
  + producer_dma_buf_fence_wait_max         // DMA_BUF_IOCTL_SYNC wait
  + inference_worst_case                     // model P99.99 + thermal-throttle margin
  + descriptor_publish_max                   // Iceoryx2 publish
  + sidecar_fd_handshake_amortized          // ≈0 outside the first handshake
  + rt_consumer_seqlock_max_retry           // retry cap × per-attempt cost
  + rt_consumer_dma_buf_view_access_max     // ≈0 with prior pinning
```

**Measurement tools:**
- 1kHz RT jitter: cyclictest 24h + custom perf counters
- dma-buf copy count: eBPF traces of `copy_to_user` / `copy_from_user`, must remain at 0
- Fence wait: trace `DMA_BUF_IOCTL_SYNC` enter/exit
- Inference worst-case: 10k-iteration stress per model + thermal soak (including a forced-throttle case via heat gun)
- Seqlock retry distribution: custom atomic counter

**Baseline comparison:**
- ROS2 + Cyclone DDS, ROS2 + rmw_iceoryx_cpp
- Message size, resolution, host spec, run length all pinned in the README (reproducibility)

## Reviewer-flagged Risks & Mitigations

| Risk | Mitigation |
|---|---|
| **Iceoryx2 SHM is not an FD-transport channel** | First-class SCM_RIGHTS / pidfd_getfd sidecar. Metadata and FDs travel on separate channels |
| **AMD XDNA external dma-buf import path unclear** | Spike PoC in week 1-2. On failure, switch the target to NVIDIA Jetson Orin |
| **Apple Silicon end-to-end zero-copy is unrealistic** | Explicitly excluded from the first build |
| **dma-buf fence/sync semantics under-specified** | `TensorDescriptor.sync_fence_kind` field; fence FDs travel through the sidecar |
| **Page faults break 1kHz determinism** | RT consumer must use mlockall + MAP_POPULATE + a forced page-fault touch |
| **Priority inversion on the dma_resv lock** | Producer is the only side that calls dma-buf attach/detach. RT consumer never holds dma_resv |
| **Buffer lifecycle / FD leaks** | `pool_generation`-based reallocation detection. RT consumer is forbidden from calling close(2) (lifecycle owned by the non-RT side) |
| **Inference long tail (P99.99, thermal throttle)** | inference_worst_case includes a thermal margin; the stale-fallback policy keeps the RT loop safe |
| **Stale infinite retry storms the RT loop** | Seqlock retry cap + injectable fallback policy |
| **Camera frame drop / NPU timeout** | Per-subsystem timeouts + explicit stale signaling. The RT loop always has a fallback |
| **Benchmark cannot be reproduced** | Pin host spec, resolution, model, and run length in the README. CI uses a self-hosted runner for reproduction |

## Open Questions

1. **First target platform** — The week 1-2 spike: if the AMD XDNA external dma-buf import smoke test succeeds, AMD; otherwise NVIDIA Jetson Orin.
2. **When to freeze the TensorDescriptor layout** — Iceoryx2 prefers fixed-size POD. Cap dynamic shape at max rank 8 and max dim 16 to start.
3. **Time-synchronization policy** — V4L2 hardware timestamps where available, corrected against the producer publish timestamp. Multi-sensor PTP later.
4. **Language surface priority** — C++ core + Python bindings for the first release. Rust crate is Phase 2 (Iceoryx2 itself is Rust).
5. **Discrete NUMA / GPUDirect** — dma-buf alone is insufficient. PCIe P2P + GPUDirect RDMA belong to a later design phase.
6. **Async producer-consumer topology** — separate processes on the same host vs. a different host (distributed). The first build is restricted to multi-process on the same host.
